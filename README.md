# URL Shortener с аналитикой

Сервис сокращения ссылок, который переживает нагрузку на редиректах за счёт кэша и не блокирует пользователя на записи статистики.

## Что решает

Короткая ссылка должна редиректить мгновенно, даже под нагрузкой, а статистика переходов не должна замедлять сам редирект. Здесь горячие ссылки читаются из Redis (cache-aside), а клики уходят в Kafka и пишутся в БД асинхронным консьюмером — с фолбэком в Postgres, если брокер недоступен.

## Демо (после `docker-compose up`)

```bash
# создать короткую ссылку
curl -s -X POST localhost:8080/api/v1/links \
  -H 'Content-Type: application/json' \
  -d '{"url":"https://go.dev/doc/effective_go"}'
# -> {"short_code":"4c92","short_url":"http://localhost:8080/4c92", ...}

# редирект (302 на оригинал) + событие клика в Kafka
curl -i localhost:8080/4c92

# статистика: всего переходов, последний клик, разбивка по дням
curl -s localhost:8080/api/v1/links/4c92/stats
```

## Стек

- **Go 1.22**, роутер **Gin**
- **PostgreSQL** (`pgx/v5`, пул соединений)
- **Redis** (`go-redis/v9`) — cache-aside для горячих кодов
- **Kafka** (`segmentio/kafka-go`) — асинхронная запись кликов
- **Docker Compose**, миграции через **golang-migrate**

## Архитектура

```
                 ┌──────────── cache-aside ───────────┐
                 │                                     ▼
Client ─▶ Gin handler ─▶ Service ─▶ Repository ─▶ PostgreSQL
                 │            │           ▲
   redirect ◀────┘            │           │ (фолбэк-запись)
                              ▼           │
                        Kafka producer    │
                              │           │
                         [link.clicks]    │
                              ▼           │
                        Kafka consumer ───┘
                     (persist кликов в отдельной горутине)
```

Слоистая архитектура `handler → service → repository`: HTTP-детали не протекают в бизнес-логику, а бизнес-логика не знает про SQL. Kafka вынесена за интерфейс `clickPublisher`, кэш — за `cacheLayer`, поэтому сервис тестируется без инфраструктуры.

Почему так:
- **base62 от id, а не UUID** — код короткий и читаемый (`4c92` вместо 36 символов), генерируется детерминированно из `nextval` последовательности, без циклов «сгенерировал-проверил-коллизия».
- **cache-aside** — на miss читаем из БД и прогреваем Redis; на удалении инвалидируем ключ.
- **клики через Kafka** — редирект не ждёт запись в БД. Если брокер лёг, продюсер пишет клик напрямую в Postgres, статистика не теряется.

## Запуск

```bash
cp .env.example .env      # для локального go run; в compose переменные уже заданы
docker compose up --build
```

Поднимаются Postgres, Redis, Kafka, применяются миграции, стартует API на `:8080`. Проверка живости: `GET /health`.

## API

| Метод | Путь | Описание |
|---|---|---|
| POST | `/api/v1/links` | создать ссылку (`url`, опц. `custom_code`, `expires_at`) |
| GET | `/{code}` | редирект `302` + событие клика в Kafka |
| GET | `/api/v1/links/{code}/stats` | всего переходов, последний клик, последние 7 дней |
| DELETE | `/api/v1/links/{code}` | удалить ссылку и инвалидировать кэш |
| GET | `/health` | health-check |

Коды ответов: `201` создано, `302` редирект, `404` нет ссылки, `409` custom-код занят, `400` невалидный URL.

## Тесты

```bash
go test ./... -race
```

Покрыто: кодирование base62 (граничные значения, детерминизм, отсутствие коллизий на 10k id), валидация URL. Бизнес-логика сервиса изолирована интерфейсами и мокается без БД/Redis/Kafka.

## Компромиссы и что бы улучшил

- **Продюсер и консьюмер в одном бинарнике.** Сделано ради простоты запуска. В проде консьюмер — отдельный сервис/деплой, чтобы масштабировать запись кликов независимо от API.
- **At-least-once доставка кликов.** Возможны дубли клика при ретраях. Для точной аналитики добавил бы идемпотентный ключ события и дедуп на консьюмере.
- **Одна партиция по умолчанию.** Ключ сообщения = `short_code`, так что при шардировании порядок по конкретной ссылке сохранится, но топик под нагрузку надо конфигурировать явно.
- **Счётчик id предсказуем.** Для публичного сервиса добавил бы «соль»/хешид, чтобы коды нельзя было перебирать инкрементом.
- **Нет rate limiting** — он есть в проекте 3 (Gateway), сюда специально не тянул, чтобы не переусложнять «разогревочный» проект.
